## 一句话理解

```
传统工程：人类写代码 → 机器执行
Harness Engineering：人类设计约束 → 智能体写代码 → 机器执行
```

**核心转变：工程师的产出从代码变成了约束系统**

---

## 核心理念

### 三层约束体系

| 层次 | 名称 | 做什么 |
|------|------|--------|
| **内层** | Prompt Engineering | 怎么跟 AI 说话 |
| **中层** | Context Engineering | 给 AI 什么信息 |
| **外层** | Harness Engineering | 让 AI 在什么环境里工作 ← **当前重点** |

### 六大核心概念

| # | 概念 | 核心原则 |
|---|------|---------|
| C1 | 地图而非手册 | 渐进式披露，入口简洁，细节在子文档 |
| C2 | 机械化执行 | lint 规则 = 修复指令，规则必须可执行 |
| C3 | 熵管理 | 文档自动生成，消除手工维护造成的漂移 |
| C4 | Agent 可读性 | 根文件 ≤60 行，约束越严自主性越强 |
| C5 | 吞吐量改变合并 | 小步快跑，背压 = 质量门禁 |
| C6 | Harness 精确定义 | Guide×Sensor 矩阵，计算性 + 推理性双重验证 |

## 二、实操手册

### 场景 1：在项目中应用 Harness 约束

**什么时候用：** 接手新项目、发现 AI 重复犯错、代码质量下降

**步骤：**

1. **建立入口 AGENTS.md（Map）**
   - 路径：`项目/AGENTS.md`
   - 内容：≤60 行，指向子文档
   - 原则：入口简洁，细节在子地图

2. **编写 validate 脚本（Linter）**
   - 路径：`项目/scripts/validate.sh`
   - 内容：检查文件结构、命名规范、必含字段
   - 执行：`bash scripts/validate.sh`

3. **编写自动生成脚本（熵管理）**
   - 路径：`项目/scripts/generate-agents.sh`
   - 内容：自动生成 AGENTS.md 内容，防止漂移

4. **精简根 AGENTS.md（≤60 行）**

5. **建立 HARNESS.md 状态文档**
   - 记录当前约束系统状态

### 场景 2：用 Harness 提升 AI 输出质量

**什么时候用：** AI 输出不稳定、格式混乱、重复犯错

**应用方法：**

| 问题 | Harness 解法 |
|------|------------|
| AI 不遵守格式 | 写 format linter，输出错误就中断 |
| 忘记关键步骤 | 写 check linter，检查每步是否完成 |
| 文档过期 | 用脚本自动生成，禁止手动编辑关键段落 |
| 规则被遗忘 | 把规则编码进 linter，不只是写文档 |

### 场景 3：建立项目知识库

**什么时候用：** 项目复杂、多个子目录、需要渐进式加载知识

**原则：**
- Slack/Google Docs/脑子里的知识 = 对 AI 不可见
- 一切决策和规范必须以版本化工件提交到仓库
- 建立多级 AGENTS.md 导航系统

## 五、快速参考

### Agent 可读性规则

- 根 AGENTS.md ≤ 60 行
- 禁止自动生成根文件（手动维护保证质量）
- 子目录 AGENTS.md 说明用途 + 下一步导航

### 熵管理规则

- 文档会腐烂，lint 规则不会
- 用脚本生成代替手动编写
- 定期扫描偏差，更新质量评分

### 吞吐量理念

- 小步快跑，合并门控最小化
- 测试偶发失败 → 重跑解决
- 背压 = 质量门禁（通过后才合并）
